# Relocalization Strategy 2026

Date: 2026-07-01

Branch: `dev/better_loop`

## Executive conclusion

The current relocalization input assumption is too weak for reliable global relocalization in ambiguous LiDAR environments.

A stationary accumulated cloud at a single stop can work in distinctive open scenes, but it should not be treated as a generally reliable global relocalization input. It lacks temporal parallax, trajectory direction, overlap evolution, and sequence-level disambiguation. In repeated indoor geometry, tunnels, gates, corridors, or partially dynamic scenes, a stationary query can produce plausible descriptor matches and plausible ICP fits at the wrong place.

The design can still be saved without replacing the whole backend:

1. Keep RHPD / Scan Context as candidate generators, not final authorities.
2. Add measurement-only comparison between stationary query descriptors and short motion-submap descriptors.
3. Add an optional query local-submap builder for relocalization and loop verification.
4. Add CPU-light geometric verification before accepting relocalization or committing loop edges.
5. Keep the existing n3mapping backend for now.
6. Treat GLIM / LIO-SAM-like systems as optional future map sources, not as immediate replacements.

The immediate goal is to reduce uncertainty. Do not do another backend rewrite before measuring whether temporal/motion submaps materially improve candidate recall and verification stability.

## Phase 1: Recent method survey, 2025-2026

This survey intentionally focuses on recent LiDAR place recognition, loop closure, relocalization, submap matching, dynamic robustness, and global consistency. Older baselines such as original LIO-SAM, GLIM, STD, and Scan Context are background only.

### 1. Efficiently Closing Loops in LiDAR-Based SLAM Using Point Cloud Density Maps, 2025

Source: https://arxiv.org/abs/2501.07399

- Input representation: local maps generated by aggregating consecutive LiDAR scans; density-preserving BEV projection; ORB descriptors; HBST-style binary retrieval; RANSAC geometric validation.
- Input unit: moving local maps, not a single stationary scan.
- CPU-friendly: yes relative to neural methods; uses BEV images, ORB, binary search tree, ground alignment, and geometric validation.
- Learning/GPU: no learning requirement in the core pipeline.
- Scope: loop closure detection, place recognition, geometric validation, multi-session alignment.
- Insight for n3mapping: this is highly relevant. It supports the hypothesis that local maps/submaps are a better recognition unit than single scans. It also explicitly handles perceptual aliasing with self-similarity pruning and uses ground alignment before BEV matching.

### 2. PRISM-Loc: Lightweight Long-range LiDAR Localization with Topological Maps, 2025

Source: https://arxiv.org/abs/2506.15849

- Input representation: topological map for global place recognition plus local pose estimation inside the selected place; 2D features and point-based optimization.
- Input unit: place/topological node plus local pose estimation, not raw global dense PCD search.
- CPU-friendly: likely yes; the paper targets lightweight long-range localization.
- Learning/GPU: no explicit GPU-learning dependency in the abstracted method.
- Scope: global localization / long-range relocalization.
- Insight for n3mapping: separate coarse place recognition from local pose estimation. Dense global maps are not required for all localization decisions; a structured map resource with local verification is enough.

### 3. RE-TRIP: Reflectivity Instance Augmented Triangle Descriptor, 2025

Source: https://arxiv.org/abs/2505.16165

- Input representation: geometric triangle descriptor augmented with LiDAR reflectivity/intensity; keypoint extraction; key instance segmentation; reflectivity-combined loop verification.
- Input unit: point cloud with local geometric/reflectivity structure.
- CPU-friendly: more CPU-friendly than deep networks, but key-instance extraction may be engineering-heavy.
- Learning/GPU: no large neural model is central, but instance extraction details need review before adoption.
- Scope: 3D place recognition and loop verification.
- Insight for n3mapping: intensity/reflectivity can help when geometry is degenerate or repeated. This may be a future optional descriptor channel if the robot's LiDAR intensity is stable. It should not be the next step because n3mapping first needs stronger query input and verification metrics.

### 4. Loop Closure via Maximal Cliques in 3D LiDAR-Based SLAM, 2026

Source: https://arxiv.org/abs/2603.05397

- Input representation: binary 3D descriptors and a compatibility graph of feature correspondences.
- Input unit: feature correspondences between scans/local maps.
- CPU-friendly: likely moderate; maximal clique search can be expensive but is deterministic and designed for real-time loop validation.
- Learning/GPU: no GPU-learning dependency indicated.
- Scope: geometric loop closure validation, replacing RANSAC-style verification.
- Insight for n3mapping: this supports moving from single ICP score to compatibility/consensus validation. However, n3mapping should not add a heavy clique solver before measuring simpler overlap and submap effects.

### 5. From Single Scan to Sequential Consistency: TempLoc, 2026

Source: https://arxiv.org/abs/2602.03198

- Input representation: sequential LiDAR scans with global coordinate estimation, correspondence prediction, uncertainty-guided coordinate fusion.
- Input unit: temporal sequence.
- CPU-friendly: no; this is a learning-based relocalization framework and likely GPU-oriented.
- Learning/GPU: yes.
- Scope: LiDAR relocalization.
- Insight for n3mapping: although the implementation is outside current constraints, the central thesis is directly relevant: single-frame relocalization is fragile, and temporal consistency improves robustness in ambiguous/dynamic scenarios.

### 6. Uni-Mapper, 2025

Source: https://arxiv.org/abs/2507.20538

- Input representation: multi-session / multi-modal LiDAR maps; voxel-wise free-space hash map for dynamic object removal; dynamic-aware loop closure; anchor-node pose graph optimization.
- Input unit: map/submap resources across sessions.
- CPU-friendly: partly; dynamic free-space hashing can be CPU-friendly, but full multi-map merging is larger than n3mapping's immediate need.
- Learning/GPU: not fundamentally GPU-learning-based from the abstract.
- Scope: dynamic-aware multi-session map merging and global consistency.
- Insight for n3mapping: dynamic filtering should be part of the serialized map resource and descriptor source, not only visualization. n3mapping already added dynamic artifact filtering; next step is to measure whether query-vs-map static consistency improves verification.

### 7. LEMON-Mapping, 2025

Source: https://arxiv.org/abs/2505.10018

- Input representation: multi-session point cloud maps, loop constraints, spatial bundle adjustment over overlapping maps.
- Input unit: overlapping submaps/maps rather than isolated pairwise edges.
- CPU-friendly: likely heavier than n3mapping's target.
- Learning/GPU: not primarily a learning method.
- Scope: globally consistent multi-session map merging.
- Insight for n3mapping: pairwise pose-graph loop edges are not enough to preserve local geometric consistency. This aligns with n3mapping's recent evidence that single-pair ICP is the wrong authority boundary.

### 8. BEV-LIO(LC), 2025

Source: https://arxiv.org/abs/2502.19242

- Input representation: BEV images from LiDAR; lightweight CNN descriptors; FAST keypoints; RANSAC coarse transform; ICP refinement.
- Input unit: LiDAR BEV images and geometric registration.
- CPU-friendly: partially; CNN descriptor may violate the pure CPU/lightweight preference unless small enough and optional.
- Learning/GPU: yes, at least for the CNN descriptor.
- Scope: LIO with loop closure.
- Insight for n3mapping: BEV projection plus coarse 2D feature verification is useful, but the CNN part is not currently aligned with constraints. A non-learning density-map variant is more appropriate.

### 9. HOTFormerLoc, 2025

Source: https://arxiv.org/abs/2503.08140

- Input representation: octree-based multi-scale point-cloud transformer.
- Input unit: point cloud / place descriptor.
- CPU-friendly: no, not as a deployed quadruped CPU-only component.
- Learning/GPU: yes.
- Scope: place recognition, especially cross-source ground/aerial and diverse environments.
- Insight for n3mapping: modern top-performing place recognition methods use richer multi-scale descriptors. But this is not the right immediate engineering path under CPU-only constraints.

### 10. ImLPR, 2025

Source: https://arxiv.org/abs/2505.18364

- Input representation: LiDAR converted to range-image views, DINOv2-based descriptor with adapters.
- Input unit: point cloud converted to image-like views.
- CPU-friendly: no.
- Learning/GPU: yes, foundation-model based.
- Scope: LiDAR place recognition.
- Insight for n3mapping: confirms the field is moving toward richer learned representations for hard retrieval, but this conflicts with the current lightweight CPU-only target.

### 11. LXD-SLAM, 2026

Source: https://arxiv.org/abs/2606.27811

- Input representation: LiDAR-centered multi-sensor fusion; continuous GP sub-meshes; Extended Scan Context; hybrid pose graph.
- Input unit: dense/submesh map representation and multi-modal measurements.
- CPU-friendly: unlikely as a minimal quadruped deployment path.
- Learning/GPU: not necessarily a learning descriptor, but the map representation and fusion stack are large.
- Scope: full dense SLAM.
- Insight for n3mapping: mature backends are moving toward richer map representations, but replacing n3mapping wholesale is not justified before measuring the query-input and verification bottleneck.

### Survey synthesis

The recent literature does not support relying on a single stationary scan/submap as a universal global relocalization input. The recurring patterns are:

- use local maps or submaps rather than single scans;
- perform candidate retrieval and geometric verification separately;
- add self-similarity / ambiguity handling before graph commit;
- use temporal or multi-frame evidence when scenes are repeated or dynamic;
- treat learned methods as stronger retrieval models, not substitutes for geometric verification;
- preserve global consistency by rejecting weak loop constraints before they enter the graph.

For n3mapping, the best-aligned near-term path is CPU-light submap-based query construction plus explicit geometric verification. A backend replacement is premature.

## Phase 2: Repository audit

This audit is based on the current `dev/better_loop` state at commit `43f1b52` and the preserved behavior baseline `c141b0e`.

### Keyframe creation

Keyframes are created in `N3MappingCore::processMappingFrame()` when `KeyframeManager::shouldAddKeyframe()` passes. The cloud and odometry pose are stored in `KeyframeManager::addKeyframe()`. The same frame then adds Scan Context and RHPD descriptors and graph odometry/prior factors.

Relevant files:

- `src/core/n3mapping_core.cpp`
- `src/keyframe_manager.cpp`
- `include/n3mapping/keyframe_manager.h`

### Cloud accumulation and submaps

There are several forms of cloud aggregation:

- `KeyframeManager::buildLocalSubmap(center_id, submap_size)` builds a world-frame local map around a keyframe.
- `buildSubmapInRootFrame(center_id, range, root_id)` builds a local submap in another keyframe's root frame.
- `buildCausalSubmapInRootFrame(center_id, range, root_id)` builds a causal submap using only past frames.
- RHPD map descriptors are saved from causal submaps in `MapSerializer`, using `rhpd_submap_kf_radius` and `rhpd_submap_voxel_size`.

This is good: the map side already supports submap descriptors. The query side is weaker.

### Query cloud for relocalization

`WorldLocalizing::buildRelocQueryCloud()` currently implements stationary/static aggregation. It keeps recent query frames only if their relative odometry motion is small enough:

- `reloc_static_agg_enable`
- `reloc_static_agg_max_frames`
- `reloc_static_agg_min_frames`
- `reloc_static_agg_max_translation`
- `reloc_static_agg_max_rotation`
- `reloc_static_agg_voxel_size`

Default config shows this is a near-stationary aggregator: max translation is 0.25 m and max rotation is 0.20 rad. This helps denoise a stopped scan, but it is not a motion-built local submap. It deliberately rejects the temporal parallax that would help disambiguate repeated geometry.

### Scan Context and RHPD descriptor computation

Loop descriptors:

- Scan Context descriptors are made in `LoopDetector::makeScanContext()` and added through `addDescriptor()`.
- RHPD descriptors are computed through `LoopDetector::computeRHPD()` / `RHPDManager`.
- RHPD uses ring-height/planar components, negative space, vertical tokens, and PCA confidence when enabled.

Relocalization descriptors:

- `WorldLocalizing::searchCandidates()` computes RHPD and optionally Scan Context from the current query cloud.
- It searches the map RHPD manager first.
- It may append frame-level RHPD candidates from a separate frame RHPD index.
- It falls back to Scan Context if needed.

### Relocalization candidate retrieval and verification

`WorldLocalizing::relocalize()` uses a two-stage structure:

1. Descriptor retrieval through `searchCandidates(query_cloud)`.
2. Basin grouping in map XY space.
3. Per-basin verification with `evaluateSingleCandidate()`.
4. Multi-window hypothesis tracking with log likelihood, winner streak, margin, and ratio gates.

This is a better structure than direct descriptor acceptance. However, the query cloud used for retrieval and verification is still usually a stationary aggregate, not a motion submap.

`evaluateSingleCandidate()` verifies candidates by aligning the query cloud against a map-side local submap built around the candidate keyframe. It tries yaw hypotheses from SC or configured yaw samples, then chooses the best converged fitness result.

### Loop candidate generation and acceptance

Loop candidates are produced by `LoopDetector::detectLoopCandidates()` using RHPD first, then Scan Context fallback. Spatial candidates are merged in `N3MappingCore::processPendingLoopClosures()`.

The current preserved `prediction_range_gate` rejects candidates before expensive submap ICP when the optimized query/match prediction exceeds `loop_max_range`. This is a valid engineering fix because it prevents far candidates from entering expensive ICP.

Current loop verification path:

1. Descriptor/spatial candidates are merged.
2. Prediction range gate runs before ICP.
3. Query source submap is built with `buildSubmapInRootFrame(query_id, 0, match_id)`.
4. Match target submap is built with `buildSubmapInRootFrame(match_id, gicp_submap_size, match_id)`.
5. `LoopVerifier::verifyPreparedSubmaps()` runs ICP from identity correction in match frame.
6. Verification requires convergence, fitness, inlier, and geometry gates.
7. Heightmap diagnostics and vertical diagnostics may be logged.
8. Segment/referee logic evaluates remaining loops.
9. Graph trial/shadow diagnostics run before commit as final damage guard.
10. `LoopClosureManager::applyEdges()` commits loop edges to `GraphOptimizer`.

The main fragility is asymmetry: the query side entering loop ICP is still a single keyframe cloud in the match frame, while the target is a multi-keyframe submap. This increases the chance that a repeated local fragment looks good under ICP.

### Graph optimization commit

`LoopClosureManager::buildLoopEdges()` builds loop edges and `applyEdges()` adds them to `GraphOptimizer`. `GraphOptimizer::incrementalOptimize()` optimizes pending factors and commits them only after successful optimization.

This commit flow is reasonable. The issue is not primarily the graph optimizer implementation; it is whether a loop measurement is graph-worthy before commit.

### Z residual and loop diagnostics

The repository already logs rich loop diagnostics:

- measured/predicted/correction transforms;
- candidate residuals;
- Z overlap ratios;
- robust Z spans;
- source/target Z centroid delta;
- vertical information ratio;
- vertical hypothesis diagnostics;
- heightmap overlap and ground dz statistics;
- segment consistency;
- graph trial diagnostics;
- accepted-loop CSVs and matrix summaries.

Recent falsifiers show that single vertical/heightmap/graph-trial signals are not clean enough as standalone gates. They should remain diagnostics and guard evidence, not final loop authority.

### Dense trajectory representation

Dense trajectory metadata is serialized in pbstream. `global_map.pcd` is visualization only. This is the right separation: localization/relocalization should depend on the serialized map/keyframes/descriptors/graph, not the visualization PCD.

### Fragile assumptions

1. Stationary aggregation is not the same as a local submap.
2. RHPD/SC retrieval is still too easy to over-trust in repeated geometry.
3. Single-pair ICP can produce plausible but graph-damaging constraints.
4. Query loop source cloud is weaker than map-side target submap.
5. Graph trial is useful as a damage guard but failed as candidate selector/authority.
6. Debug fields are strong, but the system still lacks a controlled A/B metric comparing stationary-query descriptors against motion-submap descriptors.

## Phase 3: Engineering diagnosis

### 1. Is stationary accumulated cloud at one stop enough?

Blunt answer: no, not as a general assumption.

It can work when the place is distinctive, the local geometry has enough structure, dynamic objects are minimal, and the viewpoint is close to map keyframes. It is not sufficient for reliable global relocalization in repeated indoor geometry or visually/geometrically aliased areas.

A stationary aggregate improves density and suppresses scan noise, but it does not add the missing information needed to disambiguate similar places. Motion does. A short motion submap gives:

- temporal parallax;
- trajectory direction;
- evolving overlap pattern;
- more complete spatial context;
- a stronger basis for overlap and residual-distribution checks.

### 2. Minimum stronger input

The minimum stronger input should be an optional short query local submap, built over either time or distance:

- temporal window: several seconds or several incoming frames;
- distance window: about 2-5 m of trajectory for a quadruped-scale platform;
- max frames/keyframes to bound CPU cost;
- voxel downsampled output;
- gravity/yaw prior retained from odometry/IMU if available.

The correct minimum is not a long sequence model. It is a short, CPU-light local submap that gives enough parallax to distinguish repeated geometry.

### 3. What role should RHPD play?

RHPD should be candidate generator plus source of yaw/descriptor prior. It should not be final relocalization authority.

Recommended role:

```text
RHPD / SC -> candidate generation and ranking
local submap verification -> pose hypothesis quality
multi-frame consistency -> lock decision
trial/shadow graph -> final damage guard for loop edges
```

RHPD can remain central to retrieval, but descriptor score alone must not trigger relocalization lock or loop edge commit.

### 4. CPU-light verification to add

Add verification in layers, cheapest first:

1. Voxel downsample query and map submaps.
2. Compute predicted overlap / occupied-cell overlap before expensive ICP.
3. Use gravity/yaw prior where available.
4. Run coarse ICP / NDT / GICP only on top-K candidates.
5. Log inlier ratio, residual distribution, transform magnitude, and transform stability.
6. Reject high Z inconsistency and poor overlap before accepting lock/loop commit.
7. For loop edges, keep trial optimization as final damage guard; never commit a loop that worsens trial consistency.

Important: Z residual should not be a descriptor retrieval criterion. It belongs in geometric verification and graph-commit safety.

### 5. Should mapping backend be replaced by GLIM / LIO-SAM-like backend?

No full replacement now.

Practical answer:

- Keep the n3mapping backend for now.
- Improve query input and verification first.
- Add optional external-map import later.
- Do not make GLIM/LIO-SAM the default mapping backend until evidence shows n3mapping odometry/graph is the dominant failure, not loop/relocalization candidate authority.

A mature backend may provide better odometry and map quality, but it will not solve repeated-geometry place recognition by itself. Bad loop constraints can damage any backend.

## Phase 4: Minimal implementation roadmap

The roadmap is staged to avoid large risky changes.

### Stage A: Measurement only

Goal: compare stationary-query descriptors against motion-submap descriptors without changing runtime behavior.

Add metrics:

- stationary query cloud frame count, motion span, point count;
- candidate recall@K for stationary query descriptor;
- candidate recall@K for motion-submap query descriptor;
- overlap and geometric verification success/failure by candidate source;
- Z residual and overlap metrics before graph commit;
- JSON/CSV outputs for offline matrix comparison.

Suggested file-level changes:

- `include/n3mapping/relocalization_debug_logger.h`
  - Add query cloud diagnostic fields to `RelocalizationDebugEvent`.
- `src/relocalization_debug_logger.cpp`
  - Serialize query mode, frame count, motion span, raw/downsampled point counts, descriptor mode, recall-evaluation placeholders.
- `src/world_localizing.cpp`
  - Add measurement-only construction of a comparison query submap when debug/eval mode is enabled.
  - Do not use the comparison submap to accept/reject relocalization yet.
- `tools/n3mapping_loop_debug_analyze.py` and/or a new `tools/n3mapping_reloc_debug_analyze.py`
  - Summarize recall@K, geometric verification pass/fail, and lock outcomes.
- `tools/n3mapping_eval_matrix.py`
  - Add relocalization query-mode columns if relocation artifacts are present.

Safety rule:

- Stage A must not alter default behavior.
- If the implementation cannot be compiled and tested locally, do not land it.

Implementation status:

- Implemented as debug-only instrumentation in `WorldLocalizing`.
- Runtime relocalization still uses the existing stationary/static query cloud.
- When `reloc_debug_enable=true`, `relocalization_debug.jsonl` now records both:
  - `query_*`: current stationary/static query mode, frame count, motion span, raw/downsampled points, and normal candidate count;
  - `motion_query_*`: measurement-only short motion-submap query stats and candidate count/top candidates.
- The motion-submap query is not used for lock, reject, ICP acceptance, or map state updates.

### Stage B: Query local submap construction

Goal: add an optional query submap builder.

Required modes:

- stationary accumulation mode: current behavior;
- short temporal window mode;
- distance-based window mode;
- max frame/keyframe count;
- voxel downsampling.

Suggested file-level changes:

- Add a small `RelocQuerySubmapBuilder` or keep it inside `WorldLocalizing` first if minimal.
- Extend `Config` with disabled-by-default fields only after Stage A proves usefulness.
- Keep current behavior default.

Suggested config fields later:

```cpp
bool reloc_query_submap_enable = false;
std::string reloc_query_submap_mode = "stationary"; // stationary | temporal | distance
int reloc_query_submap_max_frames = 10;
double reloc_query_submap_max_distance = 5.0;
double reloc_query_submap_max_duration = 3.0;
double reloc_query_submap_voxel_size = 0.12;
```

### Stage C: Descriptor and retrieval

Goal: allow RHPD / Scan Context descriptors to be computed from the query submap.

Rules:

- Descriptor retrieval remains candidate generation only.
- Do not directly accept based on descriptor score.
- Keep map-side descriptors serialized in pbstream.
- Compare query descriptor modes in eval before default changes.

File-level changes:

- `src/world_localizing.cpp`: compute descriptors from selected query cloud/submap.
- `src/RHPDescriptor.cpp`: no immediate change unless metrics show descriptor instability.
- `src/map_serializer.cpp`: no immediate change; map-side RHPD already uses causal submaps.

### Stage D: Lightweight geometric verification

Goal: add CPU-light verification after descriptor retrieval.

Reject candidates with:

- poor occupied-cell overlap;
- high residual;
- inconsistent Z;
- unstable transform;
- low inlier ratio.

Implementation order:

1. Add overlap metric from voxelized query/map submap.
2. Add residual distribution metric after coarse registration.
3. Add Z-consistency gate only after it cleanly separates failures in Stage A/B evidence.
4. Keep all rejected candidates in debug logs.

### Stage E: Loop/relocalization acceptance policy

Goal: conservative acceptance.

Policy:

```text
descriptor candidate must exist
AND geometric verification must pass
AND Z / height consistency must pass
AND trial optimization must not worsen global consistency
```

For loops:

- never commit a loop edge that fails shadow/trial verification;
- do not use graph trial as the retrieval source or selector;
- keep it as final damage guard.

For relocalization:

- relocalization lock requires temporal hypothesis consistency, not a single descriptor/ICP hit.

### Stage F: Optional external backend compatibility

Do not replace the backend now.

Define later import compatibility for mature backends:

- imported keyframe poses;
- keyframe clouds or submap clouds;
- odometry edges if available;
- optional loop edges;
- dense trajectory metadata;
- descriptor regeneration from imported map resource.

Keep n3mapping focused on:

- map resource handling;
- dynamic/static filtering;
- descriptor retrieval;
- relocalization;
- loop verification;
- evaluation and diagnostics.

1. Add measurement fields for relocalization query cloud type.
2. Add offline comparison tooling for stationary vs motion-submap candidate recall.
3. Add JSON/CSV summaries that can be compared across datasets.

### After measurement evidence

1. Add optional query motion submap builder.
2. Use it only when explicitly enabled.

Required windows:

- KITTI360 drive0005 holdover;
- at least one additional KITTI360 drive;

- accepted false loop count;

## Blunt answers

### Is single stationary accumulation likely sufficient?

No. It is useful as a denoised query, but it is not a reliable global relocalization input in repeated or ambiguous LiDAR scenes.

### Are temporal/motion submaps required?

For robust general relocalization, yes. The minimum stronger input is a short local submap built from several frames or a few meters of motion, not a long learned sequence model.

### Should RHPD be final decision or candidate generator?

Candidate generator only, optionally with yaw/descriptor prior. Final decision requires geometric verification and temporal consistency.

### Is backend replacement necessary now?

No. Backend replacement is premature. Add optional external-map import later, but first measure whether motion-query submaps and verification fix the relocalization/loop authority problem.

### What is the current highest-risk design point?

The highest-risk point is treating a weak local query cloud plus descriptor retrieval plus single-pair ICP as enough evidence for global graph consistency. That assumption should be replaced by measured submap-based query evidence and conservative verification.
